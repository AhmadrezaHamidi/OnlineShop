۱. ساختار JWT Token و نقش‌ها

بعد از تأیید OTP، یک JWT صادر می‌شود که نوع کاربر و نقش‌ها در آن قرار دارد. هر endpoint بر اساس Claim های این token دسترسی را چک می‌کند.

مشتری (Customer)
// Customer Token { "sub": "12345", "phone": "09121234567", "type": "Customer", "roles": [], "exp": 1234571490 }
فروشنده (Seller)
// Seller Token { "sub": "67890", "phone": "09129876543", "type": "Seller", "roles": ["Seller"], "exp": 1234571490 }
⚠️ ادمین‌ها از سیستم BackOffice جداگانه‌ای استفاده می‌کنند و به دیتابیس به صورت دستی اضافه می‌شوند. آن‌ها JWT مخصوص BackOffice با Role های مختلف دریافت می‌کنند.

۴. Flow های پرداخت
💳 پرداخت زرین‌پال (آنلاین)
۱
ثبت سفارش
POST /api/v1/orders
{ paymentMethod: "ZarinPal" }
۲
افزودن آیتم‌ها
POST /api/v1/orders/{id}/items
۳
درخواست پرداخت ZarinPal
POST /api/v1/orders/{id}/payments
→ دریافت Authority + RedirectUrl
۴
Redirect کاربر به درگاه
zarinpal.com/pg/StartPay/{authority}
۵
بازگشت به Callback
GET /payment/callback?Status=OK&Authority=xxx
۶
Verify + تأیید
PATCH /orders/{id}/payments/{pid}/complete
→ OrderStatus: Confirmed ✅
!!
پرداخت ناموفق
PATCH /orders/{id}/payments/{pid}/fail
→ OrderStatus: Pending (تلاش مجدد)
🚚 پرداخت درب منزل (COD)
۱
ثبت سفارش
POST /api/v1/orders
{ paymentMethod: "CashOnDelivery" }
۲
افزودن آیتم‌ها و ثبت نهایی
POST /api/v1/orders/{id}/items
POST /api/v1/orders/{id}/place
۳
ارسال سفارش
PATCH /api/v1/orders/{id}/ship
→ OrderStatus: Shipped
۴
تحویل به مشتری + دریافت پول
ادمین پرداخت نقدی را ثبت می‌کند
POST /orders/{id}/payments (COD)
۵
تأیید تحویل
PATCH /orders/{id}/payments/{pid}/complete
+ PATCH /orders/{id}/deliver
→ OrderStatus: Delivered ✅

۵. Flow کامل سفارش
چرخه وضعیت سفارش:
⏳ Pending → ✅ Confirmed → ⚙️ Processing → 📦 Shipped → 🏠 Delivered
هر مرحله (قبل از Delivered) می‌تواند → ❌ Cancelled    Delivered می‌تواند → ↩️ Refunded
API های هر مرحله:
وضعیتAPIعاملشرط
PendingPOST /ordersCustomerJWT Required
+ آیتمPOST /orders/{id}/itemsCustomerStatus = Pending
+ پرداختPOST /orders/{id}/paymentsCustomer—
ConfirmedPATCH /orders/{id}/payments/{pid}/completeSystem/Adminپرداخت موفق
ProcessingPATCH /orders/{id}/process (اختیاری)OrderManagerStatus = Confirmed
ShippedPATCH /orders/{id}/shipOrderManagerStatus = Confirmed
DeliveredPATCH /orders/{id}/deliverOrderManagerStatus = Shipped
CancelledDELETE /orders/{id} یا PATCH /orders/{id}/cancelCustomer / Adminقبل از Delivered
ℹ️ موجودی محصول در هر مرحله:
• AddItem: موجودی Reserve می‌شود
• RemoveItem: موجودی Release می‌شود
• Cancel: تمام موجودی‌های Reserved آزاد می‌شوند
• ConfirmStock (بعد از پرداخت): موجودی به صورت قطعی کسر می‌شود
